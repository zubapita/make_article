# Contract 08: 商用品質プロンプト改善

## 優先度: 8

## 目的

AI生成記事の品質を「商用レベル」に引き上げるため、プロンプトを改善し、複数ラウンドレビューの仕組みを整備する。
Plan.mdの「実際に商用品質の完成原稿ができるまでやる」を達成する。

## 現状の問題

- 現在のプロンプトは汎用的で、出力品質にばらつきがある
- リサーチ結果の活用指示が曖昧
- レビューが1回で終了し、品質が不十分でも完了してしまう
- 読者ターゲットや文体の指定がない

## 前提

- Contract 07（SKILL外部化）が実装済みであること
- 外部SKILLファイルを編集するだけでプロンプト改善が可能

## 変更対象ファイル

| ファイル | 変更内容 |
|---------|---------|
| `skills/opening.md` | 品質重視の質問プロンプト |
| `skills/research.md` | 情報源評価・ファクトチェック指示 |
| `skills/outline.md` | 構成の論理性・読者導線の指示 |
| `skills/draft.md` | 文体・引用・構成の品質基準 |
| `skills/review.md` | チェックリストベースのレビュー |
| `skills/redraft.md` | レビュー指摘の反映基準 |

## 実装仕様

### 1. Opening プロンプト改善

追加する質問項目:
- ターゲット読者（年齢・知識レベル・関心）
- 掲載媒体と文体トーン
- 記事の目的（情報提供 / 説得 / エンタメ / 問題提起）
- 参考にしたい記事や文体

### 2. Research プロンプト改善

追加指示:
- 情報源の信頼性評価（一次情報 / 公的機関 / 専門メディア）
- 数値データの出典明記
- 複数視点の収集（賛否両論）
- 情報の新しさの確認

### 3. Outline プロンプト改善

追加指示:
- 読者の離脱ポイント想定
- 各セクションの想定文字数
- 冒頭のフック（統計・問いかけ・エピソード）
- 結論で読者に求めるアクション

### 4. Draft プロンプト改善

追加指示:
- 一文の長さ制限（40-60文字目安）
- 段落の長さ制限（3-5文）
- 引用は原文ママ＋出典URL
- 専門用語は初出時に説明
- PREP法 or SDS法の適用

### 5. Review プロンプト改善

チェックリスト形式:
- [ ] 事実誤認・論理矛盾はないか
- [ ] 引用元は正確か
- [ ] 読者ターゲットに適した文体か
- [ ] 冒頭は読者を引き込むか
- [ ] 結論は行動を促すか
- [ ] 冗長な表現はないか

### 6. UIへのレビュー回数表示（任意）

プロジェクトページにレビュー回数を表示する。
`messages.json` 内のsystemメッセージで「レビューが完了しました。」の出現回数をカウント。

## テスト仕様

### 検証方法

この契約は主にプロンプト内容の改善であり、自動テストでの品質検証は困難。
以下の手動検証を実施する:

| # | 検証項目 | 方法 |
|---|---------|------|
| 1 | 各SKILLファイルが読み込まれる | E2Eテストで外部ファイル利用を確認 |
| 2 | プレースホルダーが展開される | ユニットテスト |
| 3 | 実際のCLIエージェントで品質確認 | 手動で1記事を作成し、品質評価 |

### 品質評価基準（手動）

- 事実の正確性: リサーチ内容と照合
- 引用の適切さ: 出典URLが機能する
- 文体の一貫性: ターゲット読者に合致
- 構成の論理性: 読後に要点が理解できる
- 読みやすさ: 一文60文字以内、段落5文以内
